# BDD-ATDD 技能
强制执行结构化的代码变更流程：先定义行为（BDD），再编写验收测试初稿（ATDD初稿）供用户审阅，
然后实现代码，接着自验证，最后根据实际验证结果完善ATDD文档。

## 何时激活

**激活条件：**
- 用户请求代码修改（实现、修复、重构、添加、更新、变更）
- 用户描述了对现有代码的需求或行为变更
- 用户要求修改正确性至关重要的源代码

**不激活条件：**
- 纯信息查询（"X 是如何工作的？"、"解释这段代码"）
- 非代码任务（仅编写文档、创建不涉及行为变更的配置）
- 琐碎的单行修改（函数名调整、错别字修正、CSS 颜色修改等）
- 用户明确表示跳过 BDD/ATDD 流程

## 五阶段工作流
本工作流为**严格顺序执行**。在当前阶段完成之前，不得进入下一阶段。

### 阶段一：BDD 规约——先定义再构建

1. **分析需求**。如果用户的请求不够明确，先澄清范围。
通过提问来理解：有哪些输入、哪些输出、哪些边界情况、哪些错误情况。

2. **从需求中推导特性名称**，使用短横线命名法（kebab-case）。
- 提取核心概念（2-5 个词）
- 全部小写，空格和特殊字符替换为短横线，不使用连续短横线
- 示例："为登录添加邮箱验证" → `email-validation-login`

3. **定位或创建 PROJECT_STATE 目录**：
- 在项目根目录查找`PROJECT_STATE/`
- 如果已存在，直接使用。如果不存在，创建`PROJECT_STATE/`并包含`BDD/`和`ATDD/`子目录

4. **阅读 BDD 模板**，位于本技能目录下的`references/bdd-template.md`。

5. **创建`PROJECT_STATE/BDD/<特性名称>.md`**，包含：
- 以需求名称为标题的特性（Feature）标题
- 简要描述
- 所有场景：正常路径、边界情况、错误条件
- 每个场景使用 Given/When/Then 格式

注:如果`PROJECT_STATE/BDD`目录下已存在疑似同名的文件，先检查内容是否相关，如果相关但不完全匹配，更新该文件以包含新的场景，而不是创建新文件。

6. **将 BDD 文档提交给用户审阅**后再继续。
等待用户确认或反馈。如有需要，调整文档内容。

### 阶段二：ATDD 初稿——定义验收标准

1. **仅在用户确认 BDD 文档之后**进行。

2. **阅读 ATDD 模板**，位于本技能目录下的`references/atdd-template.md`。

3. **创建`PROJECT_STATE/ATDD/<特性名称>/`目录**。

4. **编写`PROJECT_STATE/ATDD/<特性名称>/acceptance.md`初稿**，包含：
- 对应 BDD 文档路径的引用
- 验证策略（具体日期、策略方法、验证方式）
- 对每个 BDD 场景：**计划**的测试用例，包含预期的 Given/When/Then、计划使用的测试命令
- 各测试用例的**结果**字段留空（待自验证后填写）
- 各测试用例的**证据**字段留空（待自验证后填写）
- 汇总表留空（待自验证后填写）

5. **将 ATDD 初稿提交给用户审阅**。
等待用户确认验收标准是否合理。如有需要，调整测试用例设计。

### 阶段三：实现——编写代码
1. **仅在 ATDD 初稿经用户确认之后**才开始编码。
2. 按照 BDD 场景和 ATDD 初稿中定义的验收标准实现代码变更。
3. 始终牢记所有 BDD 场景——每个场景描述了代码必须满足的行为。
4. 如果在实现过程中发现新的场景，回到阶段一，先将新场景添加到 BDD 文档中，并同步更新 ATDD 初稿，再进行实现。

### 阶段四：自验证——证明代码有效
1. **对每个 BDD 场景**，验证实现是否满足该场景。
阅读本技能目录下的`references/verification-guide.md`获取详细策略。

2. **验证优先级**（使用可获得的最高质量方法，详见 `references/verification-guide.md` 中的证据质量层级）：
   - **首选**：运行已有测试或编写并运行针对性测试 → 记录测试命令和完整输出
   - **次选**：运行验证命令、调用工具(MCP/Skill)或执行脚本 → 记录命令和结果
   - **第三选**：观察运行时行为（HTTP响应、文件变化、进程状态） → 记录观测到的结果
   - **最后手段**：代码审查并附理由——必须说明为什么无法获得可执行证据，并描述具体行为，而非仅仅给出行号范围

3. **记录结果**：对每个场景，标注通过/失败及证据。

4. **保存非文字证据**：如果自验证过程中产生了截图、日志文件等非文字证据，
将其保存到`PROJECT_STATE/ATDD/<特性名称>/`目录下（如 `evidence-tc1.png`、`evidence-tc2.log`）。

5. **如果任何场景验证失败**：
- 调查根本原因
- 修复代码
- 重新验证该场景（以及修复可能影响的其他场景）

6. **不得跳过任何场景**，也不得在没有证据的情况下将其标记为通过。

### 阶段五：ATDD 定稿——完善验收文档

1. **仅在自验证完成且所有场景通过之后**进行。

2. **更新`PROJECT_STATE/ATDD/<特性名称>/acceptance.md`**，将初稿中的空字段填充为实际结果：
- 填写每个测试用例的**结果**（通过/失败）
- 填写每个测试用例的**证据**（测试输出、日志片段等）
- 如有非文字证据文件（截图、日志等），在证据字段中使用相对路径引用：`详见 [证据文件名](./evidence-tc1.png)`
- 填写汇总表：场景 → 测试用例 → 结果
- 在备注中说明任何限制条件（如跳过的场景、无法完全验证的条件等）

3. **诚实且完整**。如果某个场景确实无法验证应如实说明。

## 快速参考——文件位置

```
PROJECT_STATE/
  BDD/
    <特性名称>.md                        # BDD规约文档（阶段一）
  ATDD/
    <特性名称>/
      acceptance.md                      # ATDD验收文档（阶段二初稿 → 阶段五定稿）
      evidence-tc1.png                   # 非文字证据文件（如截图）
      evidence-tc2.xml                   # 非文字证据文件（如日志）
```
