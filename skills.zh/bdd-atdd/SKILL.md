# BDD-ATDD 技能
强制执行结构化的代码变更流程：先定义行为（BDD），再编写验收测试初稿（ATDD初稿）供用户审阅，
然后实现代码，接着自验证，最后根据实际验证结果完善ATDD文档。

## 原则
- **需求澄清**：如果用户请求有多种理解方式，呈现它们让用户选择。如果边界条件不确定，问清楚
- **只写必要的场景**：每个 BDD 场景必须对应一个明确的需求点。不要私自添加用户没要求的场景
- **代码只改必须改的**：实现阶段只修改与当前 BDD 场景相关的代码，不趁机重构相邻代码或"改进"风格
- **证据服务于人的可信度判断**：验证时优先提供人可直接检验的证据（关键截图、实际输出），而非仅提供可执行证据（测试通过、命令成功）。当变更对人可见时，必须补充人类可直观检验的证据

## 五阶段工作流
本工作流为**严格顺序执行**。在当前阶段完成之前，不得进入下一阶段。

### 阶段一：BDD 规约——先定义再构建

**完成标准**：BDD 文档已创建，涵盖所有正常/边界/错误场景，且已获得用户确认。

1. **分析需求**。如果用户的请求不够明确，先澄清范围。
通过提问来理解：有哪些输入、哪些输出、哪些边界情况、哪些错误情况。
**停下来问的时机**：如果需求有超过一种合理的实现方式，呈现给用户选择而非自行决定。

2. **分析现有代码**。在编写 BDD 场景之前，必须先理解需求涉及区域的现有代码逻辑，确保 Given/When/Then 基于代码事实而非猜测。
   - **定位相关代码**：根据需求涉及的功能区域，找到相关的代码
   - **理解实际交互流程**：默认状态、用户的操作路径、触发条件、数据加载时机、状态切换逻辑
   - **关键原则**：Given 必须反映真实的初始状态，When 必须反映真实的操作路径。不要假设初始状态或操作一步到位——先看代码再写场景

3. **从需求中推导特性名称**，使用短横线命名法（kebab-case）。
- 提取核心概念（2-5 个词）
- 全部小写，空格和特殊字符替换为短横线，不使用连续短横线
- 示例："为登录添加邮箱验证" → `email-validation-login`

4. **定位或创建 PROJECT_STATE 目录**：
- 在项目根目录查找`PROJECT_STATE/`
- 如果已存在，直接使用。如果不存在，创建`PROJECT_STATE/`并包含`BDD/`和`ATDD/`子目录

5. **阅读 BDD 模板,提供多种编写方案给用户选择**，BDD模板位于本技能目录下的`references/bdd-template.md`。
**务必提出2-3种BDD的编写方案，让用户选择，等用户明确后再编写BDD**

6. **创建`PROJECT_STATE/BDD/<特性名称>.md`**，包含：
- 以需求名称为标题的特性（Feature）标题
- 简要描述
- 所有业务场景：从业务视角划分所有场景
- 每个场景使用 Given/When/Then 格式，**必须详细描述完整的交互过程**（操作路径、数据输入、界面/状态变化）——因为 BDD 是唯一描述功能行为的文档，ATDD 不再重复 Given/When/Then
- **必须使用业务/领域语言**——非技术人员能理解的语言，技术实现细节（命令参数、API路径、数据结构等）留给ATDD

注:如果`PROJECT_STATE/BDD`目录下已存在疑似同名的文件，先检查内容是否相关，如果相关但不完全匹配，更新该文件以包含新的场景，而不是创建新文件。

7. **将 BDD 文档提交给用户审阅**后再继续。
等待用户确认或反馈。如有需要，调整文档内容。

### 阶段二：ATDD 初稿——定义验收标准

**完成标准**：ATDD 初稿已创建，每个 BDD 场景有对应的测试用例计划，且已获得用户确认。

1. **仅在用户确认 BDD 文档之后**进行。

2. **阅读 ATDD 模板**，位于本技能目录下的`references/atdd-template.md`。

3. **创建`PROJECT_STATE/ATDD/<特性名称>/`目录**。

4. **编写`PROJECT_STATE/ATDD/<特性名称>/acceptance.md`初稿**，包含：
- 对应 BDD 文档路径的引用
- 验证策略（具体日期、策略方法、验证方式）
- 对每个 BDD 场景：**计划**的测试用例，通过"对应BDD场景"追溯来源，包含验证思路（获取证据的方法和过程）、计划使用的测试命令
- 各测试用例的**结果**字段留空（待自验证后填写）
- 各测试用例的**证据**字段留空（待自验证后填写）
- 汇总表留空（待自验证后填写）

5. **将 ATDD 初稿提交给用户审阅**。
等待用户确认验收标准是否合理。如有需要，调整测试用例设计。

### 阶段三：实现——编写代码

**完成标准**：所有 BDD 场景对应的代码变更已实现，没有遗留 TODO 或占位实现。

1. **仅在 ATDD 初稿经用户确认之后**才开始编码。
2. 按照 BDD 场景和 ATDD 初稿中定义的验收标准实现代码变更。
3. 始终牢记所有 BDD 场景——每个场景描述了代码必须满足的行为。
4. 如果在实现过程中发现新的场景，回到阶段一，先将新场景添加到 BDD 文档中，并同步更新 ATDD 初稿，再进行实现。
5. **只改必须改的**：不要趁机重构相邻代码、添加"改进"、或改变代码风格。每一行变更都应该能追溯到某个 BDD 场景。

### 阶段四：自验证——证明代码有效

**完成标准**：每个 BDD 场景都有通过/失败结论及证据，无遗漏、无无证据的"通过"。

1. **对每个 BDD 场景**，验证实现是否满足该场景。
阅读本技能目录下的`references/verification-guide.md`获取详细策略。

2. **验证优先级**（详见 `references/verification-guide.md` 中的证据可信度评估）：
   - **等级S（目标）**：自动化测试通过 **+** 人类可直观检验的证据（如：测试通过 + 改动后关键截图）
   - **等级A**：运行已有测试或编写并运行针对性测试 → 记录测试命令和完整输出
   - **等级B**：运行验证命令、调用工具(MCP/Skill)或执行脚本 → 记录命令和结果
   - **等级C**：观察运行时行为（HTTP响应、文件变化、进程状态） → 记录观测到的结果
   - **等级D（最后手段）**：代码审查并附理由——必须说明为什么无法获得可执行证据，并描述具体行为，而非仅仅给出行号范围
   - **关键**：当变更对人产生可感知影响（UI、API、数据输出等）时，**必须**补充人类可直观检验的证据，将等级提升至S

3. **记录结果**：对每个场景，标注通过/失败及证据。

4. **保存非文字证据**：对于改动后的关键截图、日志文件等人类可直观检验的证据，
保存到`PROJECT_STATE/ATDD/<特性名称>/`目录下（如 `tc1-homepage.png`、`evidence-tc2.log`）。
当变更对人可见时，应主动采集变更后的关键截图作为证据。

5. **如果任何场景验证失败**：
- 调查根本原因
- 修复代码
- 重新验证该场景（以及修复可能影响的其他场景）

6. **不得跳过任何场景**，也不得在没有证据的情况下将其标记为通过。

### 阶段五：ATDD 定稿——完善验收文档

**完成标准**：ATDD 文档中所有空字段已填充实际结果和证据，汇总表完整，无不一致之处。

1. **仅在自验证完成且所有场景通过之后**进行。

2. **更新`PROJECT_STATE/ATDD/<特性名称>/acceptance.md`**，将初稿中的空字段填充为实际结果：
- 填写每个测试用例的**结果**（通过/失败）
- 填写每个测试用例的**证据**（测试输出、日志片段等）
- 如有非文字证据文件（截图、日志等），在证据字段中使用相对路径引用：`详见 [证据文件名](./evidence-tc1.png)`
- 填写汇总表：场景 → 测试用例 → 结果
- 在备注中说明任何限制条件（如跳过的场景、无法完全验证的条件等）

3. **诚实且完整**。如果某个场景确实无法验证应如实说明。

## 快速参考——文件位置

```
PROJECT_STATE/
  BDD/
    <特性名称>.md                        # BDD规约文档（阶段一）
  ATDD/
    <特性名称>/
      acceptance.md                      # ATDD验收文档（阶段二初稿 → 阶段五定稿）
      tc1-homepage.png                   # 关键截图（用例编号+描述性名称）
      evidence-tc2.log                   # 其他证据文件（日志等）
```
