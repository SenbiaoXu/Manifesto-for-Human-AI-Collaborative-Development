# BDD 文档模板

## 用途
本模板定义了BDD（行为驱动开发）规约文档的结构。BDD文档在编写代码**之前**捕获功能的预期行为，
使用Given/When/Then格式进行描述。

## 文档结构
每个BDD文档**必须**遵循以下结构：

```markdown
# 功能：[功能名称]

## 描述
[1-3句话描述该功能的作用及其存在原因]

## 背景（可选）
[适用于该功能所有场景的共享前置条件]

Given [共享前置条件]

## 场景：[场景名称]
Given [前置条件]
And [附加前置条件]
When [操作]
Then [预期结果]
And [附加结果]

## 场景：[另一个场景名称]
...

## 边界情况

### 场景：[边界情况名称]
Given [边界情况前置条件]
When [边界情况操作]
Then [边界情况预期结果]

## 错误条件

### 场景：[错误场景名称]
Given [错误前置条件]
When [触发错误的操作]
Then [预期的错误处理]
```

## 编写指南

1. **使用业务/领域语言，避免技术实现语言**。BDD文档面向人类审阅，必须使用非技术人员（产品经理、业务分析师等）能理解的语言描述行为。技术实现细节（命令行参数、API端点路径、数据结构名称、配置项名称等）属于ATDD，不应出现在BDD中。
   - ✓ 好的写法："以无头模式（在后台）启动浏览器" — 描述行为，人一眼就懂
   - ✗ 差的写法："使用 `--headless` 参数启动浏览器" — `--headless` 是技术实现细节，属于ATDD
   - ✓ 好的写法："用户提交登录请求后，系统验证凭据并返回认证令牌" — 描述行为
   - ✗ 差的写法："POST /api/login 返回 JWT token" — API路径和数据结构是技术细节，属于ATDD
2. 覆盖所有场景：正常路径、边界情况和错误条件。
3. 每个场景应只测试一个行为。如果一个场景测试了多个内容，将其拆分为多个独立场景。
4. 使用"And"在单个场景中连接多个前置条件或结果。
5. 在"Then"子句中描述**可观测的行为结果**——避免使用"它能正常工作"这类模糊断言，应该描述具体可感知的行为变化（如"系统返回成功响应"、"用户被重定向到首页"、"页面显示错误提示"等）。精确的断言值和验证方式由ATDD定义。
6. 仅当多个场景共享相同的前置条件设置时，才使用"背景"部分。
7. **BDD与ATDD的语言分工**：
   - BDD回答"**期望什么行为**"——用业务语言描述输入、操作和预期结果
   - ATDD回答"**如何验证该行为**"——用技术语言定义具体的测试步骤、命令、断言和验证方式
   - 如果一条描述需要技术背景才能理解，它很可能属于ATDD而非BDD

## 完整示例

```markdown
# 功能：用户邮箱登录

## 描述
允许用户使用邮箱地址和密码登录系统。系统验证邮箱格式和登录凭据，登录成功后返回认证令牌。

## 背景
Given 用户注册系统正在运行

## 场景：使用有效凭据成功登录
Given 系统中已注册邮箱为 "alice@example.com" 的用户
And 该用户的密码为 "SecurePass123!"
When 该用户使用邮箱和密码提交登录请求
Then 登录成功，系统返回认证令牌
And 令牌在24小时内有效

## 边界情况

### 场景：使用格式错误的邮箱登录
When 用户使用格式无效的邮箱地址 "not-an-email" 提交登录请求
Then 系统拒绝登录并提示"邮箱格式无效"

## 错误条件

### 场景：系统服务不可用时登录
Given 登录服务依赖的数据库不可用
When 用户使用有效凭据提交登录请求
Then 系统提示"服务暂时不可用，请稍后重试"
```
