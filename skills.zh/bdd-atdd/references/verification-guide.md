# 自验证指南

## 用途

本指南帮助你在阶段四（自验证）中决定如何验证每个BDD场景。验证策略取决于项目类型、可用的测试框架以及场景的性质。

## 决策树

对每个BDD场景，按以下顺序判断：

1. **是否有已有的测试覆盖了该场景？**
- 是 → 运行已有测试。将输出记录为证据。
- 否 → 继续步骤 2。

2. **该场景是否可以自动化测试？**
- 是 → 编写测试。运行测试。记录输出。
- 否 → 继续步骤 3。

3. **该场景是否可以通过阅读代码和追踪逻辑来验证？**
- 是 → 执行代码审查验证。记录检查了哪些内容。
- 否 → 将该场景标记为"无法验证"并说明原因。

## 检测测试框架
检查项目中是否声明了具体的测试框架、是否有保持测试用例特征的文件夹。

## 针对性测试
如果有测试框架，应在不运行完整测试套件的情况下验证特定场景。
1. 在项目的测试目录中创建测试文件（遵循已有的命名规范）。
2. 测试名称应与BDD场景对应（如 `test_valid_login_returns_jwt`）。
3. 测试结构为：设置Given、执行When、断言Then。
4. 运行测试并捕获输出。
5. 在ATDD文档中记录精确的测试命令。

如果没有测试框架，应考虑使用以下策略：
1. 是否有合适的校验命令可辅助验证
2. 是否有合适的工具(MCP/Skill)可用于验证
3. 是否可临时编写脚本进行验证

## 代码审查验证策略

### 代码路径追踪
1. 阅读相关的源代码。
2. 针对场景的输入追踪执行路径。
3. 验证每个分支和条件是否符合预期结果。
4. 记录：`通过追踪代码路径验证：[文件]:[行号] 至 [文件]:[行号]`

### 输出检查
1. 使用场景的输入运行程序。
2. 捕获标准输出/标准错误或API响应。
3. 与预期结果进行比对。
4. 记录：`通过运行 [命令] 验证，输出匹配 [预期模式]`

### 配置审查
1. 阅读与场景相关的配置文件。
2. 验证配置项是否符合预期行为。
3. 记录：`通过审查 [路径] 处的配置验证：[键]=[值] 符合预期行为`

## 处理验证失败

如果某个场景验证失败：

1. 在ATDD文档中如实记录失败。
2. 调查根本原因。
3. 修复代码。
4. 重新验证该场景。
5. 使用新结果更新ATDD文档。

**不得跳过任何场景，也不得在没有证据的情况下将其标记为通过。**

## 处理无法验证的场景

某些场景在当前环境下无法验证（如需要特定硬件、外部服务或生产数据）。在这种情况下：

1. 在ATDD文档中将该场景标记为"未验证"。
2. 说明无法验证的原因。
3. 描述理想情况下应如何验证（理想的测试配置）。
4. 记录任何可以做到的部分验证。
