# 自验证指南

## 用途

本指南帮助你在阶段四（自验证）中决定如何验证每个BDD场景。验证策略取决于项目类型、可用的测试框架以及场景的性质。

## 证据可信度评估

证据的质量不取决于获取方式的自动化程度，而取决于**人能否据此快速、独立地判断结果的可信度**。
一条高质量证据必须同时满足两个维度：
1. **机器可执行验证**：证明代码实际行为符合预期（测试通过、命令成功等）
2. **人可直接检验**：人无需阅读代码即可直观确认结果是否正确（截图、前后对比、实际输出等）

### 证据质量层级

| 等级 | 证据类型 | 人类可验证性 | 说明 |
|---|---|---|---|
| **S** | 多维交叉验证 | 最高 | 自动化测试通过 **+** 人类可直观感知的对比证据。例如：E2E测试通过 **+** 改动前后截图对比；单元测试通过 **+** 输入/输出对照 |
| **A** | 可执行输出 | 高 | 测试运行器输出、命令退出码、工具调用结果。但人无法直观判断测试本身是否有效 |
| **B** | 可观测行为 | 良 | HTTP响应、文件系统变化、进程状态、API调用结果 |
| **C** | 产物检查 | 可接受 | 构建输出、生成的文件、配置快照 |
| **D** | 仅代码审查 | 弱 — 最后手段 | 阅读源代码并推理逻辑 |

**核心原则**：
- **证据服务于人的可信度判断**：选择证据类型时，始终问自己——"用户看到这条证据，能否不读代码就判断结果是否正确？"如果不能，补充人类可感知的证据
- **等级 A 不等于足够好**：一个通过的测试只能证明"有测试通过了"，不能证明"测的是正确的东西"。当变更对人有可见影响时（UI、API行为、输出格式等），**必须**追求等级 S——提供人类可直接检验的对比证据
- **证据必须证明代码实际行为符合预期**：而不仅仅是代码看起来正确。"我追踪了第42-58行，逻辑看起来没问题"永远不是合格证据

### 代码审查证据何时可接受

代码审查（等级D）**仅**在以下情况可接受：
- 项目没有测试框架，且无法调用验证命令或工具，也无法编写脚本
- 场景纯粹是声明性的（如配置文件结构、静态资源放置）

如果必须使用代码审查证据，你**必须**说明：
1. 为什么无法获得更强的证据
2. 你验证了什么具体行为以及如何追踪的

## 决策树

对每个BDD场景，按以下顺序判断：

1. **是否有已有的测试覆盖了该场景？**
   - 是 → 运行已有测试。记录**完整的测试输出**（命令+结果）作为证据。
   - 否 → 继续步骤 2。

2. **该场景是否可以自动化测试？**
   - 是 → 编写针对性测试。运行测试。记录测试命令和输出。
   - 否 → 继续步骤 3。

3. **该场景是否可以通过运行命令、工具或脚本来验证？**
   - 是 → 运行它。捕获输出或可观测结果作为证据。
   - 否 → 继续步骤 4。

4. **该场景是否可以通过观察运行时行为来验证？**
   - 是 → 运行程序、调用API或触发动作。捕获响应或可观测状态。
   - 否 → 继续步骤 5。

5. **仅代码审查**（等级D — 最后手段）。
   - 记录你检查了什么以及为什么无法获得更强的证据。
   - 如果连代码审查都不可行 → 将该场景标记为"无法验证"并说明原因。

6. **[所有步骤后必做] 该变更是否对人有可见/可感知的影响？**
   - 是 → **必须**额外采集人类可直观检验的证据（截图、前后对比、实际输出示例等），将证据等级提升至 S。参见下方"按变更类型的人类可验证证据"
   - 否 → 当前证据已足够

### 按变更类型的人类可验证证据

当变更对人产生可感知影响时，**必须**额外提供人类可直观检验的证据。不同类型的变更需要不同的人类可验证证据：

| 变更类型 | 必要的人类可验证证据 | 说明 |
|---|---|---|
| UI/页面变更 | 改动前后截图对比 | 人一眼就能判断UI是否如预期变化，验证测试是否真的测对了 |
| API行为变更 | 实际的请求/响应示例 | 展示具体的请求和完整响应体，而非仅"测试通过" |
| 数据/逻辑变更 | 输入/输出对照示例 | 修改前：输入X→输出Y；修改后：输入X→输出Z |
| 样式/外观变更 | 视觉对比截图 | 修改前后的渲染结果截图 |
| 配置/结构变更 | 变更前后的文件diff | git diff 输出或文件内容对比 |
| 构建/部署变更 | 构建产物或运行状态截图 | 服务启动成功、页面可访问等直观证据 |

**为什么需要人类可验证证据**：Agent 编写的测试用例可能有以下问题——
- 测试断言太弱，实际没有验证关键行为（如 `assert true`）
- 测试测的是错误的东西（如验证了按钮存在但没验证按钮文字）
- 测试与需求不匹配（需求说改颜色，测试却只检查元素存在）

人类可验证证据让用户**不读代码就能判断测试是否有效**，这是证据可信度的最终保障。

## 检测测试框架
检查项目中是否声明了具体的测试框架、是否有保持测试用例特征的文件夹。

## 针对性测试
如果有测试框架，应在不运行完整测试套件的情况下验证特定场景。
1. 在项目的测试目录中创建测试文件（遵循已有的命名规范）。
2. 测试名称应与BDD场景对应（如 `test_valid_login_returns_jwt`）。
3. 测试结构为：设置Given、执行When、断言Then。
4. 运行测试并捕获输出。
5. 在ATDD文档中记录精确的测试命令。

如果没有测试框架，应按以下顺序考虑：
1. 是否有合适的校验命令可用（如 `curl`、`docker`、构建命令）
2. 是否有合适的工具(MCP/Skill)可用于验证
3. 是否可临时编写脚本进行验证（如shell脚本、Python一行命令）

## 基于执行的验证策略

### 测试执行
1. 运行测试命令（如 `pytest tests/test_login.py::test_valid_login -v`）。
2. 捕获包含通过/失败状态的完整输出。
3. 记录：`通过测试验证：[命令] → [通过/失败及输出摘要]`

### 命令/工具调用
1. 执行相关命令或调用相关工具。
2. 捕获退出码、标准输出、标准错误或工具的响应。
3. 记录：`通过运行 [命令] 验证 → [退出码，输出片段]`

### 运行时行为观察
1. 如需要则启动程序或服务。
2. 触发场景的动作（如发送HTTP请求、调用函数）。
3. 捕获响应或可观测的状态变化。
4. 记录：`通过 [动作] 验证 → [观测到的结果，如 "HTTP 200，响应体包含token"]`

### 构建/编译验证
1. 运行构建或编译命令。
2. 检查与变更代码相关的错误或警告。
3. 记录：`通过构建验证：[命令] → [成功/失败，相关输出]`

## 代码审查验证（仅作为最后手段）

**仅在没有任何基于执行的验证方法可用时使用。** 代码审查不能证明行为——它只展示了意图。

### 当你必须使用代码审查时
1. 阅读相关的源代码。
2. 识别场景输入对应的具体逻辑路径。
3. 验证每个分支和条件是否符合预期结果。
4. 必须附带理由记录：
   ```
   通过代码审查验证（无可执行验证手段）：
   无法获得更强证据的原因：[说明]
   追踪的逻辑：[描述具体的行为路径，而非仅仅是文件:行号范围]
   关键断言：[确认了哪些具体的条件/分支]
   ```

### 反模式 — 不应记录为证据的内容
- `通过追踪代码路径验证：file.js:10 至 file.js:35` — 行号范围不是证据。
- `代码看起来正确` — 模糊且不可验证。
- `函数X处理了这种情况` — 没有演示的断言。

### 好的代码审查证据是什么样的
- `函数 validateEmail() 拒绝输入 "not-an-email"，因为正则表达式 [pattern] 不匹配——该正则定义在 config.js:15，应用于 validator.js:42。项目没有测试框架，且该验证逻辑是纯函数无I/O操作。`
  注意：即使在这种情况下，也包含了**没有运行测试的理由**，并描述了**具体行为**，而不仅仅是行号。

## 处理验证失败

如果某个场景验证失败：

1. 在ATDD文档中如实记录失败。
2. 调查根本原因。
3. 修复代码。
4. 重新验证该场景。
5. 使用新结果更新ATDD文档。

**不得跳过任何场景，也不得在没有证据的情况下将其标记为通过。**

## 处理无法验证的场景

某些场景在当前环境下无法验证（如需要特定硬件、外部服务或生产数据）。在这种情况下：

1. 在ATDD文档中将该场景标记为"未验证"。
2. 说明无法验证的原因。
3. 描述理想情况下应如何验证（理想的测试配置）。
4. 记录任何可以做到的部分验证。
