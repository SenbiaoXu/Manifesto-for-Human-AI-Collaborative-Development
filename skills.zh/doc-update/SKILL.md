# 文档更新技能
在完成所有代码改动后，基于本次改动内容和用户对话，更新项目相关文档:
1.记录整个项目演进状态(PROJECT_STATE/index.md)
2.介绍功能及使用指南(README.md)
3.沉淀利于后续演进的知识(LEARNING/index.md)

# 原则
- **只改需要改的**：只更新因本次代码变更而变得过时的文档内容，不趁机"顺手"改写、重排版或润色未受影响的段落
- **匹配已有风格**：更新文档时沿用已有文档的写作风格、格式和详细程度，不要因个人偏好改变文档的呈现方式
- **不确定就询问**：如果无法判断某文档是否需要更新，应将更新和不更新的理由告诉用户，咨询用户进行确认

## 第一步：确定变更范围
在使用本技能前，必须先通过以下方式明确本次变更：
1. **检查对话上下文**：回顾本次对话中完成了哪些代码改动
2. **使用 git diff**：运行 `git diff --stat` 查看变更文件列表，结合对话上下文确定改动范围
3. **对变更分类**：判断改动属于以下哪种类型：
   - **新增功能**：添加了新的业务功能或模块
   - **架构调整**：改变了项目结构、技术栈或架构模式
   - **功能调整**：修改了已有功能的行为
   - **缺陷修复**：修复了 bug
   - **性能优化**：优化了性能相关代码
   - **重构**：代码结构调整但行为不变
   - **文档/配置**：仅文档或配置文件改动

## 第二步：按决策矩阵更新文档

根据变更类型，决定需要更新哪些文档（必更新/按需更新/跳过）：

| 变更类型 | PROJECT_STATE/index.md 演进时间轴 | PROJECT_STATE/index.md 架构蓝图/业务模块 | PROJECT_STATE/architecture.md | PROJECT_STATE/modules/*.md | README.md | LEARNING/index.md |
|---------|:---:|:---:|:---:|:---:|:---:|:---:|
| 新增功能 | 必更新 | 按需 | 按需 | 必更新(新增模块)或按需 | 按需 | 按需 |
| 架构调整 | 必更新 | 必更新 | 必更新 | 必更新 | 按需 | 按需 |
| 功能调整 | 必更新 | 按需 | 跳过 | 必更新(涉及模块) | 按需 | 按需 |
| 缺陷修复 | 必更新 | 跳过 | 跳过 | 按需 | 跳过 | 按需 |
| 性能优化 | 必更新 | 跳过 | 按需 | 按需 | 跳过 | 按需 |
| 重构 | 必更新 | 按需 | 按需 | 按需 | 跳过 | 按需 |
| 文档/配置 | 按需 | 跳过 | 跳过 | 跳过 | 按需 | 跳过 |

**说明**：
- "必更新"：必须执行更新
- "按需更新"：仅当该文档涉及的内容确实发生了变化时才更新
- "跳过"：不需要更新
- **任何情况下演进时间轴不应跳过**（除了纯文档改动外）

## 第三步：执行文档更新

### 3.1 PROJECT_STATE/index.md
位置: `PROJECT_STATE`文件夹位于项目根目录，全局唯一。
更新：在每次完成功能开发、测试或维护任务后，必须更新`PROJECT_STATE/index.md`。
内容: 记录整个项目的演进状态，格式如下:
```PROJECT_STATE/index.md
# <一句话描述项目的核心价值与工程类型,如'基于Vue3的电商站点,采用Monorepo架构'>

# 架构蓝图
<按Markdown格式简洁描述关键技术栈及架构关系,不超过十行>
详情见:PROJECT_STATE/architecture.md

# 业务模块
<按Markdown格式简洁描述核心业务模块及能力,不超过十行>
详情见:PROJECT_STATE/modules文件夹下的具体模块文档

# 演进时间轴
- <yyyy-mm>: <三句话以内归纳上个月的改动>
- <yyyy-mm-dd-hh-mm>: <一句话描述改动(新特性/架构调整/功能调整/缺陷修复/性能优化)>
```

#### 更新规则
- **架构蓝图**: 仅当技术栈或架构关系发生变化时更新
- **业务模块**: 仅当新增/删除/大幅修改业务模块时更新
- **演进时间轴**: 每次改代码都更新,默认为 yyyy-mm-dd-hh-mm 格式，当总条目超过 20 条时，自动将早期条目归纳为 yyyy-mm 格式（归纳时保留最近一个月的详细条目）

### 3.2 PROJECT_STATE/architecture.md
仅当架构或技术栈相关变化时更新。内容范围：
- **详细技术栈清单**：分类列出(框架、路由、状态、样式、工具等)。
- **目录树结构与职责**：使用 Tree 文本格式展示核心目录，并用一句话标注每个目录的职责边界。
- **架构模式**：说明项目采用的架构模式及其分层规则、数据流向。
- **关键设计决策**：仅记录能从代码中明确推断的决策，禁止编造理由。

### 3.3 PROJECT_STATE/modules/*.md
当变更涉及某模块时，找到该模块对应的文档进行更新。查找方法：
1. 读取 `PROJECT_STATE/index.md` 中"业务模块"部分，了解模块划分
2. 根据 git diff 中变更的文件路径，匹配到对应的模块文档
3. 若变更的文件不属于任何现有模块，考虑是否需要新增模块文档或调整模块归属

模块文档**数量应内聚到5个左右,不超过10个**。每个模块文档包含：
- **业务模块划分**：按照业务/功能类别划分模块,一个模块可能包含多个子项目/文件夹。
- **核心能力**：详细说明该模块承担的具体业务/功能。
- **关键文件夹/文件**：列出该模块下最重要的 2-3 个文件夹/文件路径，并说明其内部实现的核心逻辑(切勿贴代码、函数名)。
- **依赖关系**：简要说明该模块依赖和被依赖的关键模块。

### 3.4 README.md
位置: 项目根目录的 README.md 用于整体项目介绍，每个子项目也有 README.md。
更新条件（满足任一即更新）：
- 新增了对用户可见的功能或命令
- 改变了项目的安装/启动/配置方式
- 项目介绍或功能描述已过时
内容：只应包含项目介绍及使用指南:
- **项目介绍**: 项目的背景、目标、功能、使用场景等。
- **使用指南**: 项目的主要功能及使用示例。
README.md 的内容及示例应尽量简洁。**如果只是内部代码改动，不影响用户使用方式，不要更新 README。**

### 3.5 LEARNING/index.md
位置: `LEARNING`文件夹位于项目根目录，全局唯一。
**仅在本次工作过程中出现了以下情况时才更新**（没有则跳过）：
- 执行过程中遇到了异常/错误并解决了（如构建失败、依赖冲突、运行时错误）
- 用户对 Agent 的工作方式进行了纠正或提出了改进建议
- 发现了项目中值得记录的技术约束或踩坑经验

**反模式——不应记录的内容**：
- 常规操作步骤（如"安装了依赖"）——这些不是经验
- 仅在本项目特定的命名约定——代码本身就是文档
- 未经用户确认的个人推测

内容格式:
```LEARNING/index.md
# 项目经验
以下为在此项目工作的经验, 详情只在必要时按需阅读。

# 执行异常及解决方式
1. <异常1>: <一句话描述异常及解决方式>, 详情见:<LEARNING/xxx.md>

# 来自用户的纠正
1. <纠正1>: <一句话描述用户纠正的内容>, 详情见:<LEARNING/yyy.md>
```
说明：
- 为每个执行异常及解决方式,生成对应的详情文档。
- 为每个来自用户的纠正(问题反馈、改进建议、架构要求),生成对应的详情文档。
- 对于原始的报错/异常/代码段应放到`LEARNING`文件夹下对应的详情文档中,`LEARNING/index.md`内应精准地提炼经验。
